# 公司项目看板需求文档

> 文档版本：V1.0  
> 对应产品：公司项目看板  
> 适用周期：2026 年 Q3  
> 当前载体：静态交互原型  
> 目标读者：总经办、市场、产品、增长、运营、设计、研发

## 1. 项目背景

公司同时存在多个正在立项、开发和验证的增长项目。项目信息分散在聊天、文档、表格和个人认知中，导致以下问题：

- 公司成员无法快速看清当前正在做哪些项目，以及每个项目处于什么阶段。
- 市场发现的内容 Trend 与产品功能之间缺少明确映射。
- 项目阶段推进缺少统一标准，容易出现口头确认、跳阶段或误操作。
- 流量、用户画像、转化结果和项目结论无法连续追溯。
- 看板中的部分统计值缺少数据来源和维护入口，用户难以判断其可信度。
- 总经办难以及时识别需要资源、预算或方向决策的项目。

因此需要建设一个覆盖“趋势发现—项目立项—Demo MVP—小流量验证—规模转化”的公司级项目看板。

## 2. 产品目标

### 2.1 核心目标

让公司所有成员在一个页面中清楚了解：

1. 公司目前正在推进和验证哪些项目。
2. 每个项目当前处于哪个阶段、由谁负责、下一步是什么。
3. 每个 Trend 来自哪里，映射到了哪个项目和哪个产品假设。
4. 项目的验证指标、用户画像、转化结果和结论从哪里产生。
5. 哪些项目需要跨部门协同、资源决策或风险处理。
6. 哪些项目已具备规模化增长条件。

### 2.2 Q3 业务目标

Q3 需要验证至少 3 个可规模化增长模型。单个模型必须跑通以下闭环：

`趋势需求成立 → 最小化体验有效 → 目标人群完成转化 → 留存与获客路径可复制`

建议的基础判断标准：

- 内容互动率不低于 6%。
- Demo 意向转化率不低于 5%。
- 首周留存率不低于 25%。
- 获客、转化和留存数据具备明确来源及统计口径。
- 同一方法能够复制到多个内容样本或相似人群。

以上阈值为初始值，正式系统中应支持按项目类型调整。

### 2.3 非目标

V1 暂不替代专业研发管理、财务管理和广告投放平台，不承担以下能力：

- 代码仓库、缺陷和版本发布管理。
- 完整预算审批及财务结算。
- 原始用户行为明细分析。
- 广告账户的自动投放操作。
- 员工绩效考核。

看板负责汇总项目状态、关键结论和可追溯的数据入口。

## 3. 使用角色

| 角色 | 核心诉求 | 主要操作 |
| --- | --- | --- |
| 总经办 | 看全局、识别风险、做资源决策 | 查看全局数据、审批关键节点、查看待协同事项 |
| 市场/内容 | 发现 Trend、验证内容信号和目标人群 | 新增趋势、录入来源链接、补充趋势数据、关联项目 |
| 产品 | 将趋势转成产品假设并交付 Demo | 创建项目、维护产品假设、Demo 状态和验证指标 |
| 增长/运营 | 验证人群、流量、转化和留存 | 维护画像、实验、渠道数据、转化漏斗和复盘结论 |
| 设计/研发 | 明确当前交付物和截止时间 | 查看项目上下文、领取和完成任务 |
| 公司其他成员 | 快速理解公司正在做什么 | 只读查看项目、趋势、阶段和最新进展 |

## 4. 核心业务流程

### 4.1 主流程

1. 市场发现内容 Trend，录入趋势名称、内容链接、平台、样本数据和初步人群。
2. Trend 关联现有项目，或由该 Trend 发起新项目。
3. 项目完成立项，明确项目介绍、负责人、参与人、目标用户、核心假设和首轮指标。
4. 团队验证 Trend 是否持续、目标人群是否明确、需求是否成立。
5. 产品产出最小化 Demo，并验证用户是否愿意采取目标行动。
6. 增长团队进行小流量测试，记录渠道、人群、成本和转化漏斗。
7. 团队根据数据形成继续、调整、暂停或终止结论。
8. 满足阶段门槛的项目进入规模转化，沉淀可复制的增长模型。

### 4.2 五个项目阶段

| 阶段 | 核心问题 | 必填信息/交付物 | 推进条件 |
| --- | --- | --- | --- |
| 01 立项 | 为什么做，谁负责？ | 项目名称、介绍、负责人、参与人、关联趋势、目标用户、核心假设 | 责任清晰；趋势有来源；至少存在一个可验证任务 |
| 02 趋势验证 | Trend 是短期噪声还是真实需求？ | 内容样本、平台链接、增长/互动数据、画像假设、趋势结论 | 样本和数据来源完整；需求假设通过评审 |
| 03 Demo MVP | 最小功能是否能让用户行动？ | Demo 链接/版本、功能映射、体验路径、意向转化指标 | Demo 可体验；核心行为指标达到项目阈值 |
| 04 小流量测试 | 哪类流量和人群能有效转化？ | 渠道、投放批次、画像群、预算、漏斗数据、实验结论 | 样本量达标；转化数据可追溯；形成明确结论 |
| 05 规模转化 | 能否稳定复制和放大？ | 规模化方案、单位经济模型、留存指标、负责人和资源计划 | 获客与留存模型成立；获得资源决策 |

阶段支持前进和回退，但每次变更均需填写原因并写入活动记录。

## 5. 功能需求

### 5.1 项目总览

总览页用于回答“公司正在做什么”和“项目到达了哪个阶段”。

必须展示：

- 开发中项目总数。
- 验证中 Trend 总数。
- Q3 规模化增长目标及完成进度。
- 需要协同的项目数量和原因。
- 五个阶段的项目数量分布。
- 正在开发的项目列表。
- 正在验证的 Trend 列表。

交互要求：

- 点击阶段可筛选对应项目，再次选择“全部”恢复全量。
- 支持按项目名、负责人和 Trend 搜索。
- 点击 Trend 可筛选其关联项目。
- 点击项目行打开项目详情，不离开当前总览。
- 所有统计卡片均应提供数据口径说明或可追溯入口。

### 5.2 新建项目

新建增长项目时必须录入：

- 项目名称。
- 项目介绍，包括用户问题、核心假设和预期成果。
- 初始 Trend。
- Trend 内容链接。
- 项目负责人。
- 参与人员。

创建后系统应：

- 自动将项目放入“立项”阶段。
- 自动建立项目与 Trend 的初始关联。
- 在活动记录中写入创建时间和创建人。
- 引导负责人继续补充验证指标和第一项任务。

### 5.3 Trend 管理与产品映射

Trend 不是单独的素材收藏，必须与项目和可验证假设建立关系。

每条 Trend 至少包含：

- Trend 名称。
- 来源平台和内容链接。
- 内容发布时间及采集时间。
- 播放、增长率、互动率等信号数据。
- 初步用户画像。
- 关联项目。
- 待验证的产品假设。
- 当前状态和结论。

支持：

- 一个项目关联多个 Trend。
- 一个 Trend 作为多个项目的参考，但需标记主关联项目。
- 从外部链接打开原始内容。
- 查看 Trend 数据更新时间和录入人。
- 将 Trend 映射至具体产品功能或 Demo 体验节点。

### 5.4 项目详情

项目详情采用侧边抽屉展示，包含：

- 项目介绍和当前阶段。
- 负责人、参与人及所属部门。
- Trend 来源、内容链接和验证假设。
- Trend 到产品功能的映射。
- 验证指标及其数据来源。
- 用户画像群及对应转化结果。
- 当前任务、负责人、截止时间和完成标准。
- 阶段流转操作。
- 最近活动记录。

### 5.5 验证指标

项目负责人可以新增和维护指标。每项指标必须包含：

- 指标名称。
- 当前值。
- 目标值或阶段门槛。
- 数据来源。
- 统计时间范围。
- 计算口径或说明。
- 最近更新时间和更新人。

指标按类型分为：

- 内容信号：播放量、增速、互动率、收藏率等。
- Demo 验证：访问量、完成率、意向率、核心行为完成率等。
- 流量转化：曝光、点击、落地页访问、注册、付费等。
- 用户质量：留存、复访、活跃和推荐等。
- 项目进度：Demo 完成度、用户招募进度、任务完成率等。

系统不得展示无法解释来源的数值。没有真实数据时应显示“待录入”或“预估”，不可与实测结果混用。

### 5.6 用户画像与转化

正式版需补充画像群管理，用于回答“哪些人被什么内容吸引，并在什么项目中完成转化”。

每个画像群包含：

- 画像名称和描述。
- 年龄、地区、兴趣、场景等特征。
- 来源 Trend/内容。
- 来源平台和渠道。
- 关联项目及项目类型。
- 样本量。
- 曝光、点击、体验、注册、付费、留存等漏斗数据。
- 转化结论与适用范围。

同一项目应支持对比多个画像群；同一画像群也应支持查看其在不同项目类型中的转化差异。

### 5.7 任务与协同

项目可以新增下一步任务，字段包括：

- 任务内容。
- 负责人。
- 截止日期。
- 完成标准。
- 状态：未开始、进行中、已完成、阻塞。
- 阻塞原因和所需协同方。

规则：

- 最新未完成的关键任务作为项目列表中的“下一步”。
- 逾期任务自动进入“需要协同”统计。
- 完成或重新打开任务均写入活动记录。
- 总经办可按“需要资源决策”查看待处理事项。

### 5.8 阶段流转与防误操作

阶段推进属于高影响操作，必须采用二次确认，不允许在项目列表中单击直接推进。

推进时系统应展示：

- 当前阶段和目标阶段。
- 当前阶段必须完成的检查项。
- 未完成的指标、任务或交付物。

用户需要勾选确认，必要时填写推进说明后才能提交。若关键条件不满足，系统应阻止推进或要求具备权限的角色进行例外审批。

阶段回退同样需要填写原因。所有流转记录不得被普通用户删除。

### 5.9 数据口径与活动记录

“数据口径”入口需要说明每个汇总值如何计算：

| 展示值 | 计算方式 | 维护入口 |
| --- | --- | --- |
| 开发中项目 | 未归档、未终止的项目数量 | 新建项目/项目状态 |
| 验证中 Trend | 状态为待确认或验证中的 Trend 数量 | Trend 映射与状态维护 |
| 阶段分布 | 按项目当前阶段实时汇总 | 项目阶段流转 |
| 需要协同 | 逾期任务、阻塞任务、待审批事项的去重项目数 | 任务与审批记录 |
| 验证指标 | 项目中录入的指标及其最新值 | 项目详情—验证指标 |
| Q3 模型进度 | 满足规模化标准并通过确认的模型数 | 规模化评审结果 |

活动记录至少包括新增项目、Trend 映射、指标更新、任务变化和阶段流转，并记录操作人、时间、对象、变更前后状态。

### 5.10 增长模型说明

点击“增长模型说明”或“查看验证标准”后，以弹窗展示：

- 什么是可规模化增长模型。
- 趋势需求、Demo 意向、转化和留存的判断标准。
- 当前项目是否满足各项标准。
- 数据不足或未达标项。

弹窗仅用于说明和查看，不直接修改项目状态。

## 6. 权限需求

V1 建议采用以下权限：

| 操作 | 公司成员 | 项目参与人 | 项目负责人 | 总经办/管理员 |
| --- | --- | --- | --- | --- |
| 查看项目与 Trend | 是 | 是 | 是 | 是 |
| 新建项目/Trend | 可申请 | 是 | 是 | 是 |
| 修改项目资料 | 否 | 是 | 是 | 是 |
| 维护指标与任务 | 否 | 是 | 是 | 是 |
| 推进普通阶段 | 否 | 可申请 | 是 | 是 |
| 进入规模转化 | 否 | 否 | 可申请 | 审批 |
| 恢复或删除数据 | 否 | 否 | 否 | 是 |

## 7. 数据模型

正式版至少需要以下实体：

- 项目 Project。
- 趋势 Trend。
- 趋势样本 Trend Sample。
- 产品假设 Hypothesis。
- 功能映射 Feature Mapping。
- 用户画像 Audience Segment。
- 验证实验 Experiment。
- 指标 Metric。
- 任务 Task。
- 阶段记录 Stage History。
- 活动记录 Activity Log。
- 用户与部门 User/Department。

关键关系：

- 一个项目可关联多个 Trend、画像、实验、指标和任务。
- Trend 通过产品假设及功能映射与项目建立关系。
- 指标必须归属项目，可选归属 Trend、画像或实验。
- 每次阶段变更生成一条不可覆盖的历史记录。

## 8. 状态与异常处理

- 无数据时展示明确的空状态和下一步操作入口。
- 外部链接格式不正确时阻止保存并说明原因。
- 指标缺少数据来源时不得保存。
- 项目没有负责人、核心假设或关键任务时不得推进阶段。
- 删除被其他对象引用的数据时，需要提示影响范围并二次确认。
- 网络或保存失败时保留用户输入，避免表单内容丢失。
- 多人同时编辑时提示数据已更新，并避免静默覆盖。

## 9. 非功能需求

### 9.1 易用性

- 公司成员进入首页后 10 秒内能够找到项目数量、阶段和负责人。
- 常用查看操作不超过两次点击。
- 高风险操作均有明确确认和结果反馈。
- 桌面端优先，同时适配平板和手机查看。

### 9.2 性能

- 首页首屏在正常办公网络下 2 秒内可用。
- 项目搜索和阶段筛选在 300 毫秒内反馈。
- 100 个项目、500 条 Trend 和 5,000 条活动记录下仍可正常使用。

### 9.3 安全与审计

- 正式版数据不得只存储在浏览器本地。
- 用户需登录后访问，按角色控制修改权限。
- 阶段、指标和审批记录保留操作审计。
- 外部链接打开时采用安全隔离策略。
- 支持定期备份和数据导出。

## 10. 当前原型与正式版差距

当前静态原型已经具备：

- 项目总览、阶段筛选、搜索和 Trend 筛选。
- 新建项目及项目介绍、Trend 链接、负责人和参与人录入。
- Trend 与项目映射。
- 指标、数据来源和说明维护。
- 任务、负责人、截止日期和完成标准维护。
- 阶段变更二次确认。
- 活动记录、数据口径说明和演示数据恢复。
- 浏览器本地存储。

正式系统仍需补齐：

- 登录、组织架构和角色权限。
- 服务端数据库及多人实时数据同步。
- 画像群、实验、转化漏斗和功能映射的完整数据结构。
- 阶段门槛自动检查及总经办审批。
- 消息通知、逾期提醒和待办中心。
- 数据平台、投放平台或分析工具的自动同步。
- 项目归档、终止、复盘和导出。

## 11. 版本优先级

### P0：项目流转闭环

- 项目、Trend、指标、任务和阶段历史服务端存储。
- 登录及基础角色权限。
- 阶段门槛与二次确认。
- 数据口径和操作审计。
- 项目状态、风险和协同事项。

### P1：增长验证闭环

- Trend 样本和产品功能映射。
- 用户画像群。
- 小流量实验和转化漏斗。
- 项目类型对应的动态验证标准。
- 规模化增长评审。

### P2：自动化与经营分析

- 接入内容、投放和产品分析数据源。
- 自动更新指标和异常提醒。
- 跨项目画像及转化对比。
- 周报、季度复盘和管理层经营视图。

## 12. 验收标准

首个正式版本满足以下条件即可验收：

1. 用户能创建一个信息完整的项目，并在总览中立即查看。
2. 用户能录入 Trend 来源链接，并将其关联到项目和产品假设。
3. 用户能查看每个项目当前阶段、负责人、参与人和下一步任务。
4. 用户能录入带数据来源和口径的验证指标。
5. 用户能维护至少一个画像群及其转化漏斗。
6. 项目推进前会检查阶段条件，并要求确认或审批。
7. 总览中的项目数、Trend 数、阶段数和待协同数均可追溯到明细。
8. 所有关键修改都有操作人和时间记录。
9. 不同权限角色只能执行其授权操作。
10. 多名用户在不同设备登录后能看到一致数据。

## 13. 待确认事项

- 公司项目的类型及每种类型对应的阶段指标阈值。
- “开发中项目”是否包含暂停项目，终止项目如何呈现。
- 进入“规模转化”阶段的最终审批人。
- 需要优先接入的数据平台及接口可用性。
- 用户画像是否允许保存敏感属性，以及相应的数据合规要求。
